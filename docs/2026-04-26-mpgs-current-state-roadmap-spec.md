# MPGS 当前开发进度与后续路线 Spec

- 文档日期：2026-04-26
- 项目目录：`D:/AI Coding/mpgs`
- 适用范围：当前 Tauri 桌面应用原型、Steam 数据发现链路、前端推荐浏览界面、后续迭代方向
- 文档目标：总结当前已完成能力、明确真实系统边界、给出未开发部分的推荐推进顺序

## 1. 产品目标

MPGS（现界面品牌名为 Co-Play）是一个专注于 Steam 多人联机游戏发现的桌面应用，核心目标不是做“全品类 Steam 商店镜像”，而是做一套围绕多人联机价值的发现与筛选体验：

- 面向独立多人游戏、小众精品、老牌联机游戏
- 强调“值得和朋友一起玩”的发现效率
- 将 Steam 原始数据、评论样本、游玩人数与 AI 短评结合起来
- 提供新游区、精品老游区、最近发现、收藏/愿望单/浏览记录等本地体验

## 2. 当前已完成范围

### 2.1 技术架构

当前项目已完成以下基础架构：

- 前端：React + TypeScript + Vite
- 桌面壳：Tauri 2
- 后端：Rust
- 本地数据：SQLite
- 网络访问：`reqwest`
- AI 接口：OpenAI-compatible `/v1/chat/completions`

### 2.2 已实现后端能力

当前 Rust 后端已具备：

- 本地 SQLite 初始化、迁移、默认种子游戏写入
- `get_dashboard`：读取首页数据
- `save_config`：保存 Steam Key / LLM Key / 地区 / 语言 / 模型等设置
- `sync_seed_games`：同步库内已有游戏的 Steam 详情、评论样本、当前在线人数
- `preview_steam_app_list`：预览 Steam AppList 分页结果
- `discover_steam_games`：分页扫描 Steam AppList，筛出多人游戏并导入本地库
- `assess_game_with_ai`：对单个游戏做 AI 评估并写回分数/短评
- `set_game_user_state`：更新收藏、愿望单、关注、浏览记录
- `get_user_collections`：读取用户收藏集合

### 2.3 已实现前端能力

当前前端已具备：

- 首页、多视图导航、筛选器、详情页、设置页、AI 页面、收藏页、愿望单页、浏览记录页
- 新游区展示
- 精品老游区展示
- 最近发现展示
- 右侧数据概览卡
- Steam AppList 预览
- Steam 扫描并导入多人游戏
- 配置本地保存
- AI 推荐短评触发与展示

### 2.4 当前真实数据状态

根据 2026-04-26 的本地数据库检查结果，当前系统已经不是纯种子数据：

- 本地库已有 `47` 个多人游戏
- 其中：
  - `4` 个在新游区
  - `43` 个在精品老游区
- `last_sync_at` 已更新
- `steam_discovery_last_appid` 已写入

说明当前“扫描并导入多人游戏”链路已经可以把数据写入 SQLite，只是此前首页缺少足够清晰的可见性，导致用户容易误以为没有导入成功。

## 3. 当前系统行为定义

### 3.1 数据源分层

系统当前的数据来源分为三层：

1. 本地种子游戏
   - 初始化时写入，用于应用首次可用

2. Steam 同步数据
   - 用于补充库内游戏的真实详情、评论样本、在线人数

3. Steam AppList 发现数据
   - 用于把新的多人游戏导入本地数据库

SQLite 当前已经是主数据源。前端所有展示最终都来自本地库，而不是直接依赖 Steam 响应。

### 3.2 当前分区逻辑

- 新游区：近 30 天发售的多人游戏
- 精品老游区：超过 30 天或不属于新游条件的多人游戏
- 最近发现：最近导入到本地库的游戏

注意：

- “新增到本地库”不等于“新增到新游区”
- 大量导入结果会落到精品老游区
- 因此发现报告必须区分：
  - 新增总数
  - 新游区新增数
  - 老游区新增数

### 3.3 当前推荐逻辑

推荐值当前由以下要素组合而成：

- 好评率
- 评测数量置信度
- 当前在线人数
- 多人模式匹配度
- Demo 状态
- 发售新鲜度
- AI 分数或启发式 AI 替代分

这意味着系统已经不是简单的“按在线人数排序”，而是具备基础策展逻辑。

## 4. 当前已解决的关键问题

已完成修复：

- Steam AppList 请求错误信息脱敏，避免 key 泄露
- Steam AppList 请求增加备用 host 和更清晰的错误提示
- 发现默认扫描量不再是固定 `1 页 × 12 条`
- 设置页支持配置扫描页数与每页数量
- 首页加入数据概览卡
- 首页加入最近发现区块
- 发现报告区分新游区与老游区导入量

## 5. 当前仍未完成的部分

以下能力仍未完成，且是后续产品化必须补齐的部分。

### 5.1 发现链路仍偏“手动触发”

当前发现链路虽然可用，但还是一次性手动操作：

- 用户手动点击扫描
- 手动决定扫多少页
- 无自动继续
- 无后台任务
- 无中断/恢复机制

这使其更像“开发期导库工具”，还不像正式产品中的持续发现服务。

### 5.2 缺少真正的导入可观测性

虽然现在已有最近发现和概览卡，但还没有：

- 扫描进度条
- 当前处理到的 appid
- 已处理页面数
- 本次扫描耗时
- 本次扫描明细列表
- 失败项列表与失败原因

### 5.3 多人识别策略仍偏保守

当前多人识别主要依赖详情中的多人模式字段，仍可能存在：

- 漏判：某些支持多人但分类不完整的游戏未被导入
- 误判：某些模式字段含糊的游戏被错误保留

后续需要更稳的多人识别规则，而不是只看一个字段。

### 5.4 AI 仍是“单游戏触发”

当前 AI 评估只能对单个游戏触发，未形成批量内容增强流程：

- 没有对新导入游戏批量打 AI 分
- 没有 AI 评估队列
- 没有“待评估 / 已评估 / 评估失败”状态

### 5.5 用户个性化仍很浅

当前用户状态只有：

- 收藏
- 愿望单
- 关注
- 浏览记录

但还没有：

- 基于用户行为的个性化排序
- 基于“和朋友一起玩”的组队推荐
- 标签偏好学习
- 黑名单 / 忽略游戏

### 5.6 Steam 生态联动尚未开始

当前还没有：

- Steam 愿望单导入
- Steam 账号 / profile 关联
- 最近游玩记录导入
- 好友共玩倾向分析

这会限制产品从“策展工具”升级为“个人多人游戏发现助手”。

### 5.7 筛选页仍有大量占位能力

当前前端筛选器界面已存在，但许多项仍是 UI 占位：

- 发售时间
- 语言
- 标签多选
- 价格区间
- 成人内容过滤

其中很多还没有真实查询逻辑或后端支持。

## 6. 推荐的后续开发方向

以下是推荐的迭代顺序。建议先做“发现链路产品化”，再做“AI 与个性化增强”。

### Phase 1：把发现链路做成可持续可观察的产品能力

目标：让“扫描并导入多人游戏”从开发工具升级为产品功能。

建议开发：

- 扫描任务进度显示
- 当前页 / 当前 appid / 已导入数 / 已跳过数
- 失败项记录
- 可中断与可恢复扫描
- 扫描历史记录
- 自动继续扫描直到达到目标导入数

推荐优先级：最高

### Phase 2：提高多人游戏识别质量

目标：让导入结果更准，减少漏判和噪声。

建议开发：

- 多人识别规则引擎
- 结合 categories / tags / multiplayer_modes 多信号判断
- 引入“候选但未确认”状态
- 支持人工审核或轻量白名单机制

推荐优先级：高

### Phase 3：为新导入游戏建立批量内容增强

目标：让新导入的游戏不只是“进库”，还要“可读、可筛、可推荐”。

建议开发：

- 新游戏批量 AI 评估
- 批量评论摘要
- 风险标签提取
- “适合 2 人 / 4 人 / 大团体”的场景化短评
- AI 失败回退策略

推荐优先级：高

### Phase 4：把首页推荐从“静态榜单”变成“发现工作台”

目标：增强首页的“我现在该看什么”。

建议开发：

- 最近发现区更完整的展示
- 新增“待评估”“高潜小众”“本周值得关注”分组
- 首页显式展示本次导入成果
- 根据收藏/愿望单/浏览记录做轻个性化排序

推荐优先级：高

### Phase 5：接入 Steam 用户生态

目标：让产品从公共策展升级为“与你和朋友有关”的发现工具。

建议开发：

- 导入 Steam 愿望单
- 导入最近游玩
- 根据共玩倾向生成候选
- 给愿望单中的多人游戏做变化提醒

推荐优先级：中高

### Phase 6：补齐运营与工程能力

目标：确保长期使用时不失控。

建议开发：

- 请求限流与重试
- 缓存策略
- 导库日志
- 数据库备份 / 导出
- 配置重置
- 错误诊断页

推荐优先级：中

## 7. 建议的下一次开发目标

如果只选一个最值得立刻推进的目标，建议是：

### “发现任务产品化”

最推荐先做：

- 后台扫描任务模型
- 任务进度 UI
- 可暂停 / 继续
- 扫描到指定新增数量自动停止
- 失败项日志

原因：

- 它直接决定这个应用是不是“能持续扩库”
- 它会显著降低“明明导入了但我看不出来”的体验问题
- 它是后续批量 AI、个性化推荐、Steam 愿望单导入的基础

## 8. 不建议现在优先做的方向

以下方向现在不建议排在最前面：

- 过早做复杂社交功能
- 过早做账号体系
- 先做大量视觉重构
- 先做复杂推荐模型替换

原因是：

- 当前真正限制产品价值的瓶颈，不是视觉，也不是模型，而是发现链路的持续性、可见性与可控性

## 9. 本文档结论

MPGS 当前已经完成了“从静态原型到可导库的本地多人游戏发现应用”的关键跨越。

当前最重要的事实不是“系统还很初级”，而是：

- 它已经有完整桌面壳
- 已有本地主数据源
- 已能同步 Steam 详情
- 已能从 AppList 扫描并导入多人游戏
- 已有 AI 单游戏评估
- 已有前端可浏览与本地状态系统

因此，后续开发重点应从“有没有功能”切换为“能不能持续稳定发现、展示并筛选真正值得玩的多人游戏”。

这也是下一阶段 spec 与实现计划应围绕的主轴。
